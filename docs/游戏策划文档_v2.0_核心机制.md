# 《AI Empire》游戏策划文档

## 📋 文档信息

- **项目名称**：AI Empire - AI大模型公司模拟经营游戏
- **游戏类型**：增量游戏（Idle Game）
- **目标平台**：Web浏览器
- **核心机制**：基于MMO98的盈利与玩家增长系统
- **文档版本**：v2.0
- **更新时间**：2026-03-05

---

## 一、游戏概述

### 1.1 核心概念

玩家经营一家AI大模型公司，通过购买算力设施、优化模型性能、提升服务质量来吸引用户并实现盈利。游戏的核心是管理**用户增长率**和**服务质量（Hype）**之间的平衡，在容量限制下最大化每秒盈利。

### 1.2 游戏目标

**唯一核心目标**：持续提升每秒盈利（Profit/s）

### 1.3 设计理念

- 基于真实的**用户增长动力学**：口碑传播、市场饱和、服务质量
- 容量约束的**运营挑战**：负载管理、响应延迟、Bug控制
- 长期的**策略规划**：在扩容、优化、营销之间做权衡

## 二、核心公式系统

### 2.1 盈利公式（核心）

```
每秒盈利 = 每秒用户增量 × 单价 × 收入乘数 × 双倍概率

ProfitPerSecond = DeltaUsersPerSecond × PricePerCopy × RevenueMultiplier × DoubleChance
```

**说明**：
- `DeltaUsersPerSecond`：每秒新增用户数（受增长率、Hype、市场、负载影响）
- `PricePerCopy`：每用户单价（例如API调用费用）
- `RevenueMultiplier`：收入乘数（产品质量、品牌溢价等）
- `DoubleChance`：双倍出售概率（随机事件，部分用户付费翻倍）

### 2.2 用户增长公式

```
每秒用户增量 = 用户增长率 × Hype × 市场饱和抵抗 × 负载衰减系数

DeltaUsersPerSecond = PlayersGrowthRate × Hype × MarketResistance × LoadDecay
```

#### 2.2.1 市场饱和抵抗（Market Resistance）

```
MarketResistance = 1 / (1 + (当前用户数 / 市场容量)²)

其中市场容量计算：
MarketCapacity = 基础容量 × (已发布产品数 × 续作修正) + 数据中心数 × 数据中心容量加成
```

**特性**：
- 用户数 = 0 → Resistance = 1.0（市场空白，增长最快）
- 用户数 = 市场容量 → Resistance = 0.5（增长减半）
- 用户数 = 2×市场容量 → Resistance = 0.2（增长严重放缓）
- 用户数 >> 市场容量 → Resistance ≈ 0（市场饱和）

**玩法影响**：
- 发布更多产品线（文本API、图像API、语音API）扩大市场容量
- 建造数据中心增加基础设施容量
- 避免单一产品线过度饱和

#### 2.2.2 负载衰减系数（Load Decay）

使用**SmoothStep曲线**实现平滑衰减：

```
负载率 = 实际负载 / 系统容量

当 负载率 ≤ 软容量阈值（如70%）：
    LoadDecay = 1.0（无衰减）

当 软容量 < 负载率 < 硬容量阈值（如100%）：
    t = (负载率 - 软容量) / (硬容量 - 软容量)
    LoadDecay = 1 - t² × (3 - 2t)  // SmoothStep平滑曲线

当 负载率 ≥ 硬容量阈值：
    LoadDecay = 0.0（完全阻塞，增长为0）
```

**示例**：
```
软容量 = 70%, 硬容量 = 100%

负载率 60% → LoadDecay = 1.0 (100%)
负载率 70% → LoadDecay = 1.0 (100%)
负载率 85% → LoadDecay = 0.5 (50%, 平滑衰减)
负载率 95% → LoadDecay = 0.1 (10%, 严重衰减)
负载率 100% → LoadDecay = 0.0 (0%, 完全阻塞)
```

## 三、Hype 口碑系统（核心）

### 3.1 Hype 完整计算公式

```
目标Hype = 基础Hype + 低负载奖励 - Ping影响 - Bug影响

实际Hype每帧更新：
Hype = MoveTowards(当前Hype, 目标Hype, 变化速度 × deltaTime)

最终限制在：
Hype ∈ [HypeMinimum, HypeMaximum]
```

### 3.2 Hype 影响因素详解

#### 3.2.1 基础Hype
```
ModifierType.Hype
```
- 通过升级"品牌建设"、"市场营销"等研究提升
- 基础值通常在 1.0 ～ 3.0 之间

#### 3.2.2 低负载奖励（Low Load Bonus）

```
if (负载率 < 0.5):
    低负载奖励 = +HypeLowLoadBonus
else:
    低负载奖励 = 0
```

**设计意图**：
- 服务器充裕、响应快 → 用户体验好 → 口碑提升
- 鼓励玩家保持容量冗余，不要始终满负荷运行

**典型参数**：
- `HypeLowLoadBonus` = +0.3 ～ +0.5

#### 3.2.3 Ping/延迟影响（三阶段）⚠️ 最复杂

```
Ping影响 = f(当前Ping, MinorTolerance, MajorTolerance)

阶段1: Ping ≤ MinorTolerance（低延迟区）
    Ping影响 = Lerp(+HypePingLowBonus, 0, Ping / MinorTolerance)
    // 从奖励线性递减到0

阶段2: MinorTolerance < Ping ≤ MajorTolerance（中等延迟区）
    t = (Ping - MinorTolerance) / (MajorTolerance - MinorTolerance)
    Ping影响 = Lerp(0, -HypePingMinorImpact, t)
    // 从0线性递减到惩罚值

阶段3: Ping > MajorTolerance（高延迟区）
    Ping影响 = -HypePingMinorImpact - (Ping - MajorTolerance) / HypePingMajorImpact
    // 超出容忍度后快速恶化
```

**示例（典型参数）**：
```
HypePingLowBonus = +0.2
HypePingMinorTolerance = 50ms
HypePingMajorTolerance = 150ms
HypePingMinorImpact = 0.3
HypePingMajorImpact = 100

Ping = 20ms  → 影响 = +0.12（良好体验）
Ping = 50ms  → 影响 = 0（刚好可接受）
Ping = 100ms → 影响 = -0.15（开始影响口碑）
Ping = 150ms → 影响 = -0.3（用户不满）
Ping = 250ms → 影响 = -1.3（严重负面口碑）
```

**玩法影响**：
- 购买"CDN网络"、"边缘节点"降低Ping
- 研究"延迟优化"技术
- 避免服务器过载导致响应延迟

#### 3.2.4 Bug影响

```
Bug阈值 = Bug软容量 × HypeBugBonusThreshold

if (Bug数量 ≤ Bug阈值):
    Bug影响 = 0（无影响）
else:
    超出比例 = (Bug数量 - Bug阈值) / Bug软容量
    Bug影响 = -min(超出比例, HypeBugMaximumImpact)
```

**示例**：
```
Bug软容量 = 100
HypeBugBonusThreshold = 0.8（80%阈值）
HypeBugMaximumImpact = 1.0

Bug = 50  → 影响 = 0（在容忍范围内）
Bug = 80  → 影响 = 0（刚好达到阈值）
Bug = 90  → 影响 = -0.1（开始负面影响）
Bug = 100 → 影响 = -0.2（明显影响）
Bug = 150 → 影响 = -1.0（达到最大惩罚）
```

**玩法影响**：
- 投入"测试团队"降低Bug产生率
- 研究"代码质量"技术
- 定期修复Bug（消耗资金和时间）

### 3.3 Hype 渐进变化机制

```
Hype不会瞬间变化，而是平滑移动：

每帧更新：
Hype = MoveTowards(当前Hype, 目标Hype, HypeChangeSpeed × deltaTime)
```

**设计意图**：
- 模拟口碑传播需要时间
- 避免数值突变，提供缓冲期
- 玩家需要提前规划，不能依赖临时措施

**典型参数**：
- `HypeChangeSpeed` = 0.5 ～ 1.0 / 秒
- 从1.0变化到2.0大约需要 1～2秒

### 3.4 Hype 参数汇总

| 参数 | 说明 | 典型值 | 获取方式 |
|------|------|--------|---------|
| `Hype` | 基础Hype值 | 1.0 ～ 3.0 | 研究"品牌建设"升级 |
| `HypeMinimum` | 最小值（下限） | 0.1 | 固定值 |
| `HypeMaximum` | 最大值（上限） | 5.0 | 固定值 |
| `HypeChangeSpeed` | 变化速度 | 0.5/秒 | 研究"舆情响应"升级 |
| `HypeLowLoadBonus` | 低负载奖励 | +0.3 | 固定值 |
| `HypePingMinorTolerance` | Ping低容忍度 | 50ms | 研究"延迟优化"改善 |
| `HypePingMajorTolerance` | Ping高容忍度 | 150ms | 研究"延迟优化"改善 |
| `HypePingLowBonus` | 低Ping奖励 | +0.2 | 固定值 |
| `HypePingMinorImpact` | 中等Ping惩罚 | 0.3 | 固定值 |
| `HypePingMajorImpact` | 高Ping惩罚增速 | 100 | 固定值 |
| `HypeBugBonusThreshold` | Bug惩罚触发比例 | 0.8 | 固定值 |
| `HypeBugMaximumImpact` | Bug最大惩罚 | 1.0 | 固定值 |

## 七、完整系统流程

### 7.1 每帧更新流程

```
1. 计算目标Hype
   目标Hype = 基础Hype + 低负载奖励 - Ping影响 - Bug影响

2. 更新实际Hype（平滑变化）
   Hype = MoveTowards(Hype, 目标Hype, HypeChangeSpeed × deltaTime)
   Hype = Clamp(Hype, HypeMinimum, HypeMaximum)

3. 计算市场饱和抵抗
   MarketCapacity = 基础容量 × (产品数 × 续作修正) + 数据中心数 × 中心容量
   MarketResistance = 1 / (1 + (用户数 / MarketCapacity)²)

4. 计算负载衰减
   负载率 = 实际负载 / 系统容量
   if 负载率 ≤ 软容量:
       LoadDecay = 1.0
   elif 负载率 < 硬容量:
       LoadDecay = SmoothStep平滑衰减
   else:
       LoadDecay = 0.0

5. 计算用户增长
   DeltaUsers = PlayersGrowthRate × Hype × MarketResistance × LoadDecay
   总用户数 += DeltaUsers × deltaTime

6. 计算收入
   随机双倍 = (Random() ≤ DoubleChance) ? 2 : 1
   Profit = DeltaUsers × PricePerCopy × RevenueMultiplier × 随机双倍
   资金 += Profit × deltaTime

7. 更新Ping和Bug
   根据当前负载率和系统状态动态更新
```

### 7.2 玩家策略循环

```
初期（启动阶段）：
    购买基础算力（CPU/GPU）→ 提升容量 → 承接更多用户
    
中期（优化阶段）：
    平衡三角：容量扩张 vs Hype管理 vs 产品升级
    - 负载过高 → 购买算力或优化容量利用率
    - Hype下降 → 降低Ping、修复Bug、降低负载
    - 收入不足 → 升级产品质量或提高定价
    
后期（稳态运营）：
    维持Hype在高位 → 保持负载率 < 50%
    持续升级RevenueMultiplier → 提升单用户价值
    研究自动化 → 减少手动干预
```

### 9.2 平衡原则

1. **负载率最佳区间**：40% ～ 60%
   - < 40%：容量浪费，但Hype有奖励
   - 40% ～ 60%：最佳运营状态
   - > 70%：开始衰减，Hype下降
   - > 100%：完全阻塞

2. **Hype维持策略**：
   - 保持负载率 < 50% → +0.3奖励
   - Ping < 50ms → +0.2奖励
   - Bug < 80个 → 无惩罚
   - 综合可达 Hype = 基础值 + 0.5

3. **市场饱和应对**：
   - 发布新产品线 → 扩大MarketCapacity
   - 建造数据中心 → 增加基础容量
   - 避免单一依赖

## 十、关键设计理念

### 10.1 基于MMO98的核心继承

1. **Hype作为口碑反馈系统**
   - 不是简单的固定乘数
   - 动态响应服务质量（Ping、Bug、负载）
   - 平滑变化，需要提前规划

2. **市场饱和的真实模拟**
   - 非线性阻力曲线（平方关系）
   - 通过产品创新打破天花板
   - 模拟真实市场生命周期

3. **负载管理的平滑衰减**
   - 使用SmoothStep避免陡坡
   - 70%～100%之间提供缓冲期
   - 鼓励保持容量冗余

### 10.2 与原MMO98的差异

| 对比项 | MMO98 | AI Empire |
|--------|-------|-----------|
| **背景** | 游戏运营 | AI公司运营 |
| **容量单位** | 玩家数 | 请求/秒 |
| **Ping来源** | 服务器距离 | 推理延迟 |
| **Bug来源** | 游戏漏洞 | 模型错误 |
| **产品** | 发售游戏 | 上线模型 |
| **升级** | 服务器硬件 | 算力设施 |

### A. 公式速查表

// 盈利
Profit/s = DeltaUsers/s × Price × RevMult × Double

// 用户增长
DeltaUsers/s = GrowthRate × Hype × Resistance × LoadDecay

// 市场饱和
Resistance = 1 / (1 + (Users/Market)²)

// 负载衰减（SmoothStep）
t = (Load - Soft) / (Hard - Soft)
LoadDecay = 1 - t² × (3 - 2t)

TargetHype = Base + LowLoadBonus - PingImpact - BugImpact
Hype = MoveTowards(Hype, Target, Speed × dt)
