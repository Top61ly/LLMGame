# 《AI Empire》游戏策划文档

## 📋 文档信息

- **项目名称**：AI Empire - AI大模型公司模拟经营游戏
- **游戏类型**：增量游戏（Idle Game）
- **目标平台**：Web浏览器
- **核心机制**：基于MMO98的盈利与玩家增长系统
- **文档版本**：v2.0
- **更新时间**：2026-03-08

---

## 一、游戏概述

### 1.1 核心概念

玩家经营一家AI大模型公司，通过购买算力设施、优化模型性能、提升服务质量来吸引用户并实现盈利。游戏的核心是管理**用户增长率**和**服务质量（Hype）**之间的平衡，在容量限制下最大化每秒盈利。

### 1.2 游戏目标

**唯一核心目标**：持续提升每秒盈利（Profit/s）

### 1.3 设计理念

- 基于真实的**用户增长动力学**：口碑传播、市场饱和、服务质量
- 容量约束的**运营挑战**：负载管理、响应延迟、Bug控制
- 长期的**策略规划**：在扩容、优化、营销之间做权衡

## 二、核心公式系统

### 2.1 盈利公式（核心）

```
每秒盈利 = 每秒用户增量 × 单价 × 收入乘数

ProfitPerSecond = DeltaUsersPerSecond × PricePerCopy × RevenueMultiplier
```

**说明**：
- `DeltaUsersPerSecond`：每秒新增用户数（受增长率、Hype、市场、负载影响）
- `PricePerCopy`：每用户单价（例如API调用费用）
- `RevenueMultiplier`：收入乘数（产品质量、品牌溢价等）

### 2.2 用户增长公式

```
每秒用户增量 = 用户增长率 × Hype × 市场饱和抵抗 × 负载衰减系数

DeltaUsersPerSecond = PlayersGrowthRate × Hype × MarketResistance × LoadDecay
```

#### 2.2.1 市场饱和抵抗（Market Resistance）

```
MarketResistance = 1 / (1 + (当前用户数 / 市场容量)²)

其中市场容量计算：
MarketCapacity = 基础容量 × (已发布产品数 × 续作修正) + 数据中心数 × 数据中心容量加成
```

**特性**：
- 用户数 = 0 → Resistance = 1.0（市场空白，增长最快）
- 用户数 = 市场容量 → Resistance = 0.5（增长减半）
- 用户数 = 2×市场容量 → Resistance = 0.2（增长严重放缓）
- 用户数 >> 市场容量 → Resistance ≈ 0（市场饱和）

**玩法影响**：
- 发布更多产品线（文本API、图像API、语音API）扩大市场容量
- 建造数据中心增加基础设施容量
- 避免单一产品线过度饱和

#### 2.2.2 负载衰减系数（Load Decay）

使用**SmoothStep曲线**实现平滑衰减：

```
负载率 = 实际负载 / 系统容量

当 负载率 ≤ 软容量阈值（如70%）：
    LoadDecay = 1.0（无衰减）

当 软容量 < 负载率 < 硬容量阈值（如100%）：
    t = (负载率 - 软容量) / (硬容量 - 软容量)
    LoadDecay = 1 - t² × (3 - 2t)  // SmoothStep平滑曲线

当 负载率 ≥ 硬容量阈值：
    LoadDecay = 0.0（完全阻塞，增长为0）
```

**示例**：
```
软容量 = 70%, 硬容量 = 100%

负载率 60% → LoadDecay = 1.0 (100%)
负载率 70% → LoadDecay = 1.0 (100%)
负载率 85% → LoadDecay = 0.5 (50%, 平滑衰减)
负载率 95% → LoadDecay = 0.1 (10%, 严重衰减)
负载率 100% → LoadDecay = 0.0 (0%, 完全阻塞)
```

## 三、Hype 口碑系统（核心）

### 3.1 Hype 完整计算公式

```
目标Hype = 基础Hype + 低负载奖励 - Ping影响 - Bug影响

实际Hype每帧更新：
Hype = MoveTowards(当前Hype, 目标Hype, 变化速度 × deltaTime)

最终限制在：
Hype ∈ [HypeMinimum, HypeMaximum]
```

### 3.2 Hype 影响因素详解

#### 3.2.1 基础Hype
```
ModifierType.Hype
```
- 通过升级"品牌建设"、"市场营销"等研究提升
- 基础值通常在 1.0 ～ 3.0 之间

#### 3.2.2 低负载奖励（Low Load Bonus）

```
if (负载率 < 0.5):
    低负载奖励 = +HypeLowLoadBonus
else:
    低负载奖励 = 0
```

**设计意图**：
- 服务器充裕、响应快 → 用户体验好 → 口碑提升
- 鼓励玩家保持容量冗余，不要始终满负荷运行

**典型参数**：
- `HypeLowLoadBonus` = +0.3 ～ +0.5

#### 3.2.3 Ping/延迟影响（三阶段）⚠️ 最复杂

```
Ping影响 = f(当前Ping, MinorTolerance, MajorTolerance)

阶段1: Ping ≤ MinorTolerance（低延迟区）
    Ping影响 = Lerp(+HypePingLowBonus, 0, Ping / MinorTolerance)
    // 从奖励线性递减到0

阶段2: MinorTolerance < Ping ≤ MajorTolerance（中等延迟区）
    t = (Ping - MinorTolerance) / (MajorTolerance - MinorTolerance)
    Ping影响 = Lerp(0, -HypePingMinorImpact, t)
    // 从0线性递减到惩罚值

阶段3: Ping > MajorTolerance（高延迟区）
    Ping影响 = -HypePingMinorImpact - (Ping - MajorTolerance) / HypePingMajorImpact
    // 超出容忍度后快速恶化
```

**示例（典型参数）**：
```
HypePingLowBonus = +0.2
HypePingMinorTolerance = 50ms
HypePingMajorTolerance = 150ms
HypePingMinorImpact = 0.3
HypePingMajorImpact = 100

Ping = 20ms  → 影响 = +0.12（良好体验）
Ping = 50ms  → 影响 = 0（刚好可接受）
Ping = 100ms → 影响 = -0.15（开始影响口碑）
Ping = 150ms → 影响 = -0.3（用户不满）
Ping = 250ms → 影响 = -1.3（严重负面口碑）
```

**玩法影响**：
- 购买"CDN网络"、"边缘节点"降低Ping
- 研究"延迟优化"技术
- 避免服务器过载导致响应延迟

#### 3.2.4 Bug影响

Bug影响系统是一个复杂的两阶段动态反馈机制，详见 **第四章：Bug系统**。

简要说明：
- **奖励区**：Bug数量低于阈值时，增加Hype（体现质量稳定）
- **惩罚区**：Bug数量超过阈值时，降低Hype（二次方惩罚）

**玩法影响**：
- 投入"测试团队"降低Bug产生率
- 研究"代码质量"技术
- 建造"监控系统"提升Bug容量
- 通过“校准”模块完成趣味任务，主动修复Bug

### 3.3 Hype 渐进变化机制

```
Hype不会瞬间变化，而是平滑移动：

每帧更新：
Hype = MoveTowards(当前Hype, 目标Hype, HypeChangeSpeed × deltaTime)
```

**设计意图**：
- 模拟口碑传播需要时间
- 避免数值突变，提供缓冲期
- 玩家需要提前规划，不能依赖临时措施

**典型参数**：
- `HypeChangeSpeed` = 0.5 ～ 1.0 / 秒
- 从1.0变化到2.0大约需要 1～2秒

### 3.4 Hype 参数汇总

| 参数 | 说明 | 典型值 | 获取方式 |
|------|------|--------|---------|
| `Hype` | 基础Hype值 | 1.0 ～ 3.0 | 研究"品牌建设"升级 |
| `HypeMinimum` | 最小值（下限） | 0.1 | 固定值 |
| `HypeMaximum` | 最大值（上限） | 5.0 | 固定值 |
| `HypeChangeSpeed` | 变化速度 | 0.5/秒 | 研究"舆情响应"升级 |
| `HypeLowLoadBonus` | 低负载奖励 | +0.3 | 固定值 |
| `HypePingMinorTolerance` | Ping低容忍度 | 50ms | 研究"延迟优化"改善 |
| `HypePingMajorTolerance` | Ping高容忍度 | 150ms | 研究"延迟优化"改善 |
| `HypePingLowBonus` | 低Ping奖励 | +0.2 | 固定值 |
| `HypePingMinorImpact` | 中等Ping惩罚 | 0.3 | 固定值 |
| `HypePingMajorImpact` | 高Ping惩罚增速 | 100 | 固定值 |
| `HypeBugBonusThreshold` | Bug奖励区阈值比例 | 0.3 | 固定值（已更新） |
| `HypeBugBonus` | Bug低时的Hype奖励 | +0.2 | 固定值（新增） |
| `HypeBugTolerance` | Bug容忍度（压力基准） | SoftCap × 1.0 | 动态计算（新增） |
| `HypeBugPenalty` | Bug惩罚系数 | 0.5 | 固定值（新增） |

**注意**：Bug相关参数已全面更新，详见第四章Bug系统。旧参数`HypeBugMaximumImpact`已弃用。

## 四、Bug系统

### 4.1 系统概述

Bug系统模拟AI模型在实际运营中产生的**推理错误**和**幻觉问题**。这是一个完整的动态反馈循环：

```
用户规模 + 服务器负载 → Bug持续产生 → Bug累积 → 影响Hype（两阶段）→ 影响用户增长
```

**设计理念**：
- Bug不是简单的惩罚因子，而是运营质量的动态指标
- 适度低Bug能增加Hype（质量稳定的正面口碑）
- 过高Bug会严重损害Hype（指数级恶化）
- Bug容量随用户规模动态增长，避免后期失控

---

### 4.2 Bug生成机制

#### 4.2.1 Bug生成速率公式

```
BugGenerationRate = [BaseRate + UserScaling] × LoadMultiplier × SoftCapResistance

其中：
BaseRate = ModifierType.BugsGenerationRate（基础Bug生成速率）
UserScaling = log10(max(1, 用户数)) × BugsUserScaling（用户规模影响）
LoadMultiplier = 0.5 + 负载率（负载影响，范围 0.5 ～ 1.5）
SoftCapResistance = 1 / (1 + (当前Bug数 / BugSoftCap)³)（三次方阻力）

每秒Bug累积：
Bugs += BugGenerationRate × deltaTime
```

#### 4.2.2 各因子详解

**1. 基础生成率（BaseRate）**
- 来源：`ModifierType.BugsGenerationRate`
- 受研究"代码质量"和建筑"测试团队"影响
- 典型值：0.5 ～ 5.0 个/秒

**2. 用户规模影响（UserScaling）**
```
UserScaling = log10(用户数) × BugsUserScaling

示例（BugsUserScaling = 0.1）：
用户数 = 1,000    → log10(1,000) = 3.0 → 额外生成 +0.3/秒
用户数 = 100,000  → log10(100,000) = 5.0 → 额外生成 +0.5/秒
用户数 = 10,000,000 → log10(10,000,000) = 7.0 → 额外生成 +0.7/秒
```

**设计意图**：
- 使用对数增长避免指数爆炸
- 用户越多，系统复杂度越高，Bug产生越快
- 模拟真实软件的规模效应

**3. 负载乘数（LoadMultiplier）**
```
LoadMultiplier = 0.5 + 负载率

示例：
负载率 = 0%   → 乘数 = 0.5（最低，系统空闲）
负载率 = 30%  → 乘数 = 0.8
负载率 = 70%  → 乘数 = 1.2（开始超负荷）
负载率 = 100% → 乘数 = 1.5（最高，系统满载）
```

**设计意图**：
- 服务器空闲时Bug产生率降低到基础值的50%
- 负载越高，系统压力越大，Bug产生越快
- 鼓励保持容量冗余

**4. 软上限阻力（SoftCapResistance）**
```
SoftCapResistance = 1 / (1 + (Bugs / BugSoftCap)³)

示例（BugSoftCap = 100）：
Bugs = 0   → Resistance = 1.0（无阻力）
Bugs = 50  → Resistance = 0.889（轻微阻力）
Bugs = 100 → Resistance = 0.5（减半）
Bugs = 150 → Resistance = 0.26（强阻力）
Bugs = 200 → Resistance = 0.11（极强阻力）
```

**设计意图**：
- 使用三次方阻力实现平滑衰减
- Bug接近软上限时，生成速率自动下降
- 防止Bug无限累积，提供自我调节机制

#### 4.2.3 完整示例计算

**场景**：中期运营状态
```
用户数 = 100,000
负载率 = 60%
当前Bug = 50
BugSoftCap = 100

BugsGenerationRate = 1.0（基础）
BugsUserScaling = 0.1

计算过程：
BaseRate = 1.0
UserScaling = log10(100,000) × 0.1 = 5.0 × 0.1 = 0.5
LoadMultiplier = 0.5 + 0.6 = 1.1
SoftCapResistance = 1 / (1 + (50/100)³) = 1 / (1 + 0.125) = 0.889

BugGenerationRate = [1.0 + 0.5] × 1.1 × 0.889 = 1.467 个/秒

每秒累积：Bugs += 1.467
每分钟累积：约 +88 个Bug
```

### 4.3 Bug容量系统

#### 4.3.1 动态容量公式

```
BugSoftCapacity = BugsSoftCapBase + log10(max(1, 用户数)) × BugsSoftCapScaling
BugHardCapacity = BugSoftCapacity × BugsHardCapModifier
```

#### 4.3.2 容量行为机制

**软容量（Soft Capacity）**：
- Bug接近软容量时，生成速率按三次方阻力衰减
- 不会完全阻止Bug产生，但会显著放缓
- 影响Hype的临界点基准

**硬容量（Hard Capacity）**：
- Bug达到硬容量时，**完全停止**Bug生成
- 作为绝对上限，防止数值溢出
- 典型值为软容量的2倍

#### 4.3.3 容量动态增长示例

**参数**：
```
BugsSoftCapBase = 50
BugsSoftCapScaling = 10
BugsHardCapModifier = 2.0
```

**不同阶段的容量**：

| 游戏阶段 | 用户数 | BugSoftCap | BugHardCap | 说明 |
|----------|--------|------------|------------|------|
| 初期 | 1,000 | 50 + 3×10 = 80 | 160 | 容量较小 |
| 早期 | 10,000 | 50 + 4×10 = 90 | 180 | 逐步增长 |
| 中期 | 100,000 | 50 + 5×10 = 100 | 200 | 标准容量 |
| 中后期 | 1,000,000 | 50 + 6×10 = 110 | 220 | 持续扩展 |
| 后期 | 10,000,000 | 50 + 7×10 = 120 | 240 | 大规模容量 |
| 超后期 | 1,000,000,000 | 50 + 9×10 = 140 | 280 | 顶级容量 |

**设计意图**：
- 对数增长确保容量与用户规模同步
- 避免初期Bug容量过大（降低挑战性）
- 避免后期Bug容量不足（无法正常运营）

#### 4.3.4 容量限制行为

```
if (当前Bug数 >= BugHardCapacity):
    BugGenerationRate = 0（完全停止生成）
    显示警告："系统Bug已达硬上限，无法继续累积"
else:
    正常按公式生成（包含SoftCapResistance阻力）
```

### 4.4 Bug对Hype的影响（两阶段机制）

#### 4.4.1 核心公式

```
Threshold = BugSoftCapacity × HypeBugBonusThreshold

if (Bugs ≤ Threshold):
    // 阶段1: 奖励区 - Bug少时增加Hype
    BugImpact = (1 - Bugs / Threshold) × HypeBugBonus
    
else:
    // 阶段2: 惩罚区 - Bug多时降低Hype（二次方惩罚）
    Pressure = (Bugs / HypeBugTolerance)²
    BugImpact = -Pressure × HypeBugPenalty
```

#### 4.4.2 阶段1：Bug奖励区（Bug ≤ Threshold）

**奖励阈值计算**：
```
Threshold = BugSoftCapacity × HypeBugBonusThreshold

示例（SoftCap = 100, HypeBugBonusThreshold = 0.3）：
Threshold = 100 × 0.3 = 30个Bug
```

**奖励计算**：
```
BugImpact = (1 - Bugs / Threshold) × HypeBugBonus

示例（HypeBugBonus = 0.2）：
Bugs = 0   → Impact = (1 - 0/30) × 0.2 = +0.20（完美质量）
Bugs = 10  → Impact = (1 - 10/30) × 0.2 = +0.133（优秀质量）
Bugs = 20  → Impact = (1 - 20/30) × 0.2 = +0.067（良好质量）
Bugs = 30  → Impact = (1 - 30/30) × 0.2 = 0（临界点）
```

**设计理念**：
- 低Bug代表高质量，应当获得Hype奖励
- 线性递减确保平滑过渡
- 阈值设为30%软容量（比MMO98的80%更严格）
- 使"低Bug奖励"成为需要努力维持的状态

#### 4.4.3 阶段2：Bug惩罚区（Bug > Threshold）

**压力函数**：
```
Pressure = (Bugs / HypeBugTolerance)²

说明：
- HypeBugTolerance = BugSoftCapacity × 1.0（容忍度等于软容量）
- 使用二次方函数实现加速惩罚
```

**惩罚计算**：
```
BugImpact = -Pressure × HypeBugPenalty

示例（SoftCap = 100, HypeBugTolerance = 100, HypeBugPenalty = 0.5）：

Bugs = 40  → Pressure = (40/100)² = 0.16 → Impact = -0.08
Bugs = 60  → Pressure = (60/100)² = 0.36 → Impact = -0.18
Bugs = 80  → Pressure = (80/100)² = 0.64 → Impact = -0.32
Bugs = 100 → Pressure = (100/100)² = 1.0 → Impact = -0.5（达到软容量）
Bugs = 150 → Pressure = (150/100)² = 2.25 → Impact = -1.125（严重超载）
Bugs = 200 → Pressure = (200/100)² = 4.0 → Impact = -2.0（灾难性影响）
```

**关键特性**：
- ⚠️ **二次方惩罚**：Bug超过容忍度后，Hype损失呈指数级恶化
- Bug = 软容量时：Impact = -0.5（明显惩罚）
- Bug = 2×软容量时：Impact = -2.0（4倍惩罚）
- 鼓励玩家主动控制Bug，避免接近软容量

#### 4.4.4 完整影响曲线

```
Hype Impact
    ↑
+0.20|  ●●●●●●●
     |         ●●●
+0.10|            ●●●
     |               ●●
 0.00|_________________●______________ Bugs
     |                 30 ●
-0.10|                     ●●
     |                       ●●
-0.20|                         ●●
     |                           ●●●
-0.50|                              ●●●（SoftCap=100）
     |                                 ●●●
-1.00|                                   ●●●●
     |                                       ●●●●
-2.00|                                          ●●●●●
     └────┬─────────────┬──────────┬─────────────→
       奖励区        临界点     正常区      危险区
       (0-30)       (30)     (30-100)    (>100)
```

**设计意图**：
- 奖励区窄（30%）：需要精细管理才能获得奖励
- 临界点到软容量（30-100）：渐进式惩罚，给玩家反应时间
- 超过软容量（>100）：指数级恶化，强制玩家采取行动

#### 4.4.5 与Hype系统的整合

在 **第三章 3.2 目标Hype计算** 中，Bug影响作为减项：

```
目标Hype = 基础Hype + 低负载奖励 - Ping影响 - Bug影响

注意：
- Bug影响可以为负值（实际为奖励）
- 例如 Bug影响 = -(-0.2) = +0.2 会增加目标Hype
- 当Bug ≤ Threshold时：-BugImpact = -(+0.2) = -0.2，即减去正值 = 加上负值 = 增加Hype
```

**修正后的完整公式**：
```
if (Bugs ≤ Threshold):
    目标Hype = 基础Hype + 低负载奖励 - Ping影响 + (1 - Bugs/Threshold) × HypeBugBonus
else:
    目标Hype = 基础Hype + 低负载奖励 - Ping影响 - (Bugs/Tolerance)² × HypeBugPenalty
```

### 4.5 Bug修复机制（校准模块）

本章节改为正式功能设计：在左侧导航新增“校准”栏目。玩家通过完成趣味小游戏，获得即时Bug修复量。

#### 4.5.1 模块定位

- 导航名：校准（两字）
- 设计原型：模拟大模型的纠错流程（纠正幻觉、数据标注、推理校验）
- 目标：给玩家一个可主动干预Bug曲线的轻交互入口

#### 4.5.2 小游戏池（首版）

1. 心算快答（基础数学计算）
2. 神兽验证码（图形识别与信息提取）
3. 快算24（组合推理）

> 结算统一映射为 Bug 修复量，不直接改动其他核心公式。

#### 4.5.3 修复结算公式（首版）

设本次修复值为 $R$，则：

$$
R = \text{RandomInt}(FixBugMin, FixBugMax) + StreakBonus
$$

并执行：

$$
Bugs_{new} = \max\left(0,\ Bugs_{old} - R\right)
$$

首版默认：
- $FixBugMin = 4$
- $FixBugMax = 6$
- $StreakBonus$ 初始为 0（可由连胜系统叠加）

#### 4.5.4 数学正确性校验

当 $FixBugMin=4, FixBugMax=6$，且首版 $StreakBonus=0$ 时：

$$
R \in \{4,5,6\}
$$

期望修复量：

$$
E[R] = \frac{4+5+6}{3} = 5
$$

若某时刻 $Bugs_{old}=3$，则：

$$
Bugs_{new}=\max(0,3-R)=0
$$

因此不会出现负Bug，且修复效率满足“初始每次修复约5个Bug”的设计目标。

#### 4.5.6 与核心系统耦合规则

1. 校准模块只修改 `Bugs` 资源，不直接修改 `Hype`。
2. `Bugs` 下降后，通过既有Hype公式自然体现为Bug影响减轻。

4. 任何修复效果都受下限保护：`Bugs >= 0`。

### 4.7 Bug系统的策略深度

#### 4.7.1 玩家需要平衡的要素

1. **扩张 vs 质量**
   - 快速增加用户 → Bug产生加速 → Hype下降 → 增长放缓
   - 需要在扩张前投资测试团队和代码质量

2. **负载 vs Bug控制**
   - 高负载 → Bug生成加快 + Ping上升 → 双重Hype惩罚
   - 保持40%-60%负载是最佳状态

3. **早期投资 vs 后期救火**
   - 早期不投资Bug控制 → 后期Bug失控 → 难以逆转
   - 预防性投资比应急修复更高效

#### 4.7.2 不同阶段的Bug管理策略

**初期（0-30分钟）**：
- Bug容量较小（~80），容易达到阈值
- 优先购买1个"测试团队"降低Bug产生率
- 目标：保持Bug < 24（奖励区）

**中期（30分钟-2小时）**：
- 用户快速增长，Bug产生加速
- 研究"代码质量"持续降低Bug基础生成率
- 建造"监控系统"提升Bug软容量
- 目标：保持Bug < 软容量的50%

**后期（2小时+）**：
- Bug容量随用户数增长到~120+
- 校准 + 自动化修复双轨并行
- 目标：维持稳态运营，Bug稳定在安全区

#### 4.7.3 典型错误操作

❌ **错误1**：忽略Bug累积
- 只关注扩容，不投资Bug控制
- 结果：Bug突破阈值 → Hype暴跌 → 增长停滞

❌ **错误2**：过度追求零Bug
- 过度投资Bug控制，牺牲扩张速度
- 结果：收入增长缓慢，无法进入下一阶段

❌ **错误3**：应急式管理
- 等待Bug达到硬容量才采取行动
- 结果：Hype已经严重受损，需要长时间恢复

✅ **正确策略**：
- 预防性投资：在Bug达到阈值前布局
- 动态平衡：根据当前阶段调整Bug容忍度
- 监控指标：实时关注Bug/SoftCap比率

## 八、完整系统流程

### 8.1 每帧更新流程

```
1. 计算目标Hype
   目标Hype = 基础Hype + 低负载奖励 - Ping影响 - Bug影响

2. 更新实际Hype（平滑变化）
   Hype = MoveTowards(Hype, 目标Hype, HypeChangeSpeed × deltaTime)
   Hype = Clamp(Hype, HypeMinimum, HypeMaximum)

3. 计算市场饱和抵抗
   MarketCapacity = 基础容量 × (产品数 × 续作修正) + 数据中心数 × 中心容量
   MarketResistance = 1 / (1 + (用户数 / MarketCapacity)²)

4. 计算负载衰减
   负载率 = 实际负载 / 系统容量
   if 负载率 ≤ 软容量:
       LoadDecay = 1.0
   elif 负载率 < 硬容量:
       LoadDecay = SmoothStep平滑衰减
   else:
       LoadDecay = 0.0

5. 计算用户增长
   DeltaUsers = PlayersGrowthRate × Hype × MarketResistance × LoadDecay
   总用户数 += DeltaUsers × deltaTime

6. 计算收入
   Profit = DeltaUsers × PricePerCopy × RevenueMultiplier
   资金 += Profit × deltaTime

7. 更新Ping和Bug
   根据当前负载率和系统状态动态更新
```

### 7.2 玩家策略循环

```
初期（启动阶段）：
    购买基础算力（CPU/GPU）→ 提升容量 → 承接更多用户
    
中期（优化阶段）：
    平衡三角：容量扩张 vs Hype管理 vs 产品升级
    - 负载过高 → 购买算力或优化容量利用率
    - Hype下降 → 降低Ping、修复Bug、降低负载
    - 收入不足 → 升级产品质量或提高定价
    
后期（稳态运营）：
    维持Hype在高位 → 保持负载率 < 50%
    持续升级RevenueMultiplier → 提升单用户价值
    研究自动化 → 减少手动干预
```
